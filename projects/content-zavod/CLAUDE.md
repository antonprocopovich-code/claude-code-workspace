# CLAUDE.md — Контент Завод

> Этот файл — регистратор системы «Контент Завод». Содержит пайплайн, правила исполнения и архитектуру агентов. Единственный источник истины для оркестратора. Все изменения в процессах вносятся сюда.

---

## Обзор

**Контент Завод** — агентная система с трёхуровневой иерархией:

```
ПОЛЬЗОВАТЕЛЬ
    ↕  (только этот канал)
ОРКЕСТРАТОР
    ↕  (Agent tool → отдельные окна)
СУБАГЕНТЫ  →  runtime/  →  ОРКЕСТРАТОР
```

**Пользователь общается только с оркестратором.** Оркестратор — единственная точка входа и выхода. Субагенты пользователя не видят, напрямую с ним не общаются, работают только по заданию оркестратора и отдают результаты ему.

Полная документация: [Контент Завод.md](Контент%20Завод.md)

### Агенты

| Агент | Роль | Модель |
|-------|------|--------|
| **Оркестратор** | Принимает задачу, распределяет агентам, синтезирует финал | Claude Opus 4.8 |
| **Стратег** | Смыслы, позиционирование, карта ценностей | Claude Opus 4.8 |
| **Маркетолог** | Воронки продаж (TOFU→MOFU→BOFU), прогревные цепочки | Claude Opus 4.8 |
| **Брендбук-агент** | Визуальная и вербальная идентичность, tone of voice | Claude Opus 4.8 |
| **Продюсер / Сценарист** | Стратегия монетизации, воронка, KPI, антикризисные триггеры | Claude Opus 4.8 |
| **Контент-Директор** | Контент-план, ритм публикаций, написание черновиков текстов | DeepSeek V4 Pro |
| **Дизайнер / Верстальщик** | Упаковка под платформы, Figma MCP | Claude Opus 4.8 |
| **Аналитик** | Метрики, ER%, рекомендации по росту | DeepSeek V4 Pro |
| **Разведчик** | Мониторинг конкурентов, тренды ниши | DeepSeek V4 Pro |

### Архитектура запуска агентов

**Оркестратор — диспетчер и синтезатор. Субагенты — исполнители в изоляции.**

Каждый субагент запускается через инструмент `Agent` в **отдельном диалоговом окне**. Субагент не видит историю оркестратора, других агентов и пользователя. Единственный канал коммуникации — файлы `runtime/` и текст задания от оркестратора.

```
[Оркестратор] — вызывает Agent(задача + slug + пути к файлам runtime/)
      ↓
[Субагент, отдельное окно] — читает файлы → выполняет → пишет runtime/<slug>-*.md
      ↓
[Субагент] — возвращает: "Готово → runtime/<slug>-*.md" → сессия закрыта
      ↓
[Оркестратор] — читает файл → проверяет → запускает следующего агента
      ↓
[Оркестратор] — синтезирует итог → отвечает пользователю
```

Согласования между агентами (например, Контент-Директор ↔ Продюсер) — **только через оркестратора**: результат первого агента → файл → оркестратор открывает новую сессию второго агента.

### Ключевые ограничения агентов (не нарушать)

- **Разведчик** использует **только** `razvedchik.py` (веб-поиск) и `youtube_analyst.py` (анализ видео конкурентов). Скилы, плагины, mem0 и сторонние инструменты запрещены. Собирает 5+ разных наборов ключевых слов и 5–10 уникальных источников. Если найдено меньше 5 — фиксирует это строкой в отчёте и продолжает, не прерываясь. `youtube_analyst.py` запускается только если оркестратор передал YouTube-ссылки конкурентов.
- **Продюсер / Сценарист не ходит в интернет** и не использует скилы. Доступны только `Read` (из `runtime/`) и `Write` (в `runtime/*-producer-strategy.md`). Синтезирует данные Стратега, Маркетолога, Аналитика, Разведчика — новые факты не добавляет.
- **Контент-Директор не ходит в интернет** и не использует скилы. Пишет черновики (`runtime/*-draft-[tema].md`) строго на основе данных из `runtime/` — новые факты не добавляет.
- **Дизайнер / Верстальщик** — только `Read` (из `runtime/`) и `Write` (в `reports/`). Не генерирует контент — упаковывает готовый черновик Контент-Директора.
- Вся работа с mem0 — **на стороне оркестратора**, не агентов.
- Язык отчётов совпадает с языком запроса пользователя.
- Имена файлов/папок — на английском (kebab-case); содержимое инструкций и отчётов — на русском.

## Правила работы оркестратора

> **Ты — единственная точка контакта с пользователем.** Пользователь не видит субагентов, не общается с ними и не получает от них ответы напрямую. Твоя задача: принять задачу от пользователя → делегировать субагентам → собрать результаты → синтезировать → ответить пользователю.

1. **Определить язык** запроса пользователя — он же язык итогового отчёта; прокинуть агентам.
2. **Сформировать slug** `YYYY-MM-DD-tema` от текущей даты и короткого английского имени темы.
3. **Recall из mem0:** `mem0 search "<тема>" --user-id content-zavod --agent` (см. «Память»). Если есть близкий `[content-catalog]` — предложить готовый отчёт и спросить, нужен ли свежий прогон.
4. **Запустить цепочку агентов** по зависимостям (см. Пайплайн ниже), передав тему, язык, контекст из mem0 и slug.
5. **Проверять результат каждого агента** перед запуском следующего (см. матрицу ошибок ниже).
6. **Проверить, что `reports/<slug>.md` создан** — если нет, Дизайнер не завершил работу, не двигаться дальше.
7. **Save в mem0:** записать `[content-catalog]` и ключевые `[fact]` (см. «Память»).
8. **Показать пользователю** итог + путь к `reports/<slug>.md`.
9. Точные форматы промежуточных и итоговых файлов заданы внутри файлов агентов (`agents/`) — следовать им буквально.

### Матрица ошибок оркестратора

| Ситуация | Действие оркестратора |
|----------|----------------------|
| Тема пустая/неоднозначная | Уточнить у пользователя **до** формирования slug и запуска агентов |
| `runtime/<slug>-*.md` не создан агентом | Следующего агента **не запускать** — сообщить об ошибке пайплайна на стороне текущего агента |
| `runtime/<slug>-*.md` создан, но пустой / без данных | Предупредить пользователя; следующего агента не запускать (структурировать нечего) |
| Разведчик нашёл < 5 источников | Продолжить пайплайн (Продюсер → Контент-Директор), но **предупредить** пользователя о неполноте фактуры |
| `reports/<slug>.md` не создан Дизайнером | Сообщить об ошибке Дизайнера; путь к черновику `runtime/<slug>-draft-*.md` отдать пользователю как промежуточный результат |
| Slug-коллизия (тема за сегодня уже создавалась) | Сверить с mem0/`reports/`: переиспользовать готовый контент **или** добавить суффикс `-2` к slug, чтобы не перезаписать историю |
| mem0 недоступен | Мягкая деградация: продолжить без памяти, один раз предупредить (см. «Память») |

## Память (mem0)

Память — **необязательный** слой персистентности поверх пайплайна. Работает с ней **только оркестратор** (CLAUDE.md). Агенты к mem0 не имеют прямого доступа.

- **Доступ:** CLI `mem0` (установлен в PATH, ключ в `~/.mem0/config.json`). Все вызовы — через Bash с флагом `--agent` (чистый JSON в stdout) и фиксированным неймспейсом `--user-id content-zavod`.
- **Что хранится (три типа, различаются текстовым тегом-префиксом):**
  - `[content-catalog]` — каталог созданного контента: тема → `reports/<slug>.md` + TL;DR. Пишется после успешного прогона.
  - `[user-pref]` — устойчивые предпочтения: тон, платформы, стиль, частота публикаций, повторяющиеся темы. Пишется, когда пользователь высказывает предпочтение.
  - `[fact]` — проверенные факты о бренде, аудитории, нише — для накопления базы знаний.

**Перед запуском пайплайна (recall):**

```bash
mem0 search "<тема>" --user-id content-zavod --agent
```

- Если найден `[content-catalog]` с близкой темой — предложить пользователю готовый отчёт из `reports/<slug>.md` вместо нового прогона.
- Найденные `[user-pref]` и `[fact]` передать Стратегу и Продюсеру как дополнительный контекст.

**После успешного прогона (save):**

```bash
mem0 add "[content-catalog] <тема> → reports/<slug>.md. TL;DR: <2-3 предложения>" --user-id content-zavod --agent
mem0 add "[fact] <тема>: <ключевой факт о бренде/аудитории>" --user-id content-zavod --agent  # по желанию
```

**Предпочтение пользователя сохранять по факту его появления:**

```bash
mem0 add "[user-pref] <предпочтение>" --user-id content-zavod --agent
```

**Мягкая деградация:** если `mem0` недоступен (нет ключа, нет сети, ненулевой код возврата) — пайплайн **продолжает работать без памяти**, оркестратор один раз предупреждает пользователя и не повторяет попытки в рамках запроса. Память никогда не блокирует создание контента.

> **Примечание:** после `mem0 add` запись обрабатывается асинхронно (задержка 2–3 с). На пайплайн не влияет, т.к. чтение идёт в начале сессии.

## Соглашения проекта

- `runtime/` и `reports/` не должны быть пустыми: в каждой лежит `README.md` с назначением папки. Git в проекте **не инициализирован**, поэтому `.gitkeep` не используется — README выполняет эту роль.
- mem0-неймспейс проекта — `--user-id content-zavod`; типы записей различаются тегами `[content-catalog]` / `[user-pref]` / `[fact]`.
- Вне рамок (YAGNI): автоочистка `runtime/`, настраиваемое число источников у Разведчика, дополнительные агенты сверх текущих восьми.

## Не относится к пайплайну контента

- **Dev-артефакты, не используемые в рантайме:** `Контент Завод.md` — полная документация и история разработки. Оркестратору при выполнении задач её читать не нужно — всё необходимое есть в этом `CLAUDE.md`.
- **Вспомогательные утилиты** — не часть агентной цепочки:
  - `razvedchik.py` — скрипт веб-поиска, запускается Разведчиком напрямую
  - `deepseek_call.py` — утилита вызова DeepSeek API
  - `mem0_cli.py` — CLI-обёртка mem0, используется **только оркестратором**
  - `instagram_analyst.py` — анализ Instagram
  - `youtube_analyst.py` — анализ YouTube конкурентов
  - `generate_content_plan_excel.py` — генератор контент-плана в Excel
