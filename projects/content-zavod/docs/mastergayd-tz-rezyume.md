# Резюме: Мастер-гайд по написанию ТЗ (Блок 2, Модуль 8)

> Источник: Никита Велс — курс по Claude Code / AI-автоматизации

---

## Главная мысль

ТЗ — это контракт между тобой и агентом. Без него агент додумывает детали сам. С ним — строит то, что задумано. **Качество результата агента почти никогда не упирается в модель. Оно упирается в качество входа.**

---

## Три фазы запуска проекта

```
1. DISCOVERY → понять, что на самом деле нужно (10–15+ вопросов)
2. ТЗ        → зафиксировать в одном документе-контракте (.md файл)
3. ИСПОЛНЕНИЕ → агент строит строго по ТЗ
```

Проект начинается не с кода. Код — в последней фазе.

---

## Структура хорошего ТЗ (6 областей)

| Область | Вопрос |
|---------|--------|
| 🎯 Зачем и для кого | Проблема и пользователь. Без «зачем» агент оптимизирует не то |
| 🧭 Сценарии пользователя | Что человек видит и делает шаг за шагом |
| 🗄️ Данные и состояние | Что хранить, где, временно или навсегда |
| ⚙️ Технические рамки | Язык, фреймворк, существующие системы |
| ✅ Критерии приёмки | Как понять, что готово (проверяемо, не «работает хорошо») |
| 🚧 Границы (что НЕ делаем) | Явный список «вне рамок» |

## Сырой бриф — с чего начинать

Не пиши ТЗ сразу. Сначала — короткий свободный текст:

| Пункт | Суть |
|-------|------|
| 🩹 Проблема, не решение | Что болит, 1–2 фразы |
| 👥 Для кого | Кто будет пользоваться |
| ⏱️ Почему сейчас | Что подтолкнуло |
| 🎯 Образ «хорошо» | Грубо, без деталей |
| 🔒 Жёсткие рамки | Дедлайн, бюджет, «нельзя Y» |
| 🚫 Антицели | Чего точно НЕ хочу |

**Правило:** несите проблему, не решение. «Нужно хранить заказы» — проблема. «Нужна БД» — уже решение.

## Проклятие инструкций

На 20+ пунктах в одном запросе даже сильные модели начинают «терять» половину требований.

**Вывод:** ТЗ должно быть полным, но не раздутым. Структурируй заголовками, не сваливай в абзац.

## Работа с большими ТЗ (3 подхода)

### А. Иерархия спек (parent + children)
- Один верхний файл: видение, архитектура, глоссарий, индекс
- По файлу на каждую фичу/домен
- Агент грузит верхний + только нужную дочернюю спеку

### Б. Milestone-спеки (этап за этапом)
- Каждый этап = отдельный кусок ценности
- Свой цикл: спека → план → реализация
- Только первый этап проработан детально, остальные — потом

### В. Агент-декомпозитор
- Одна большая спека → отдельный агент разбивает на атомарные задачи
- Каждая задача: что сделать, критерий приёмки, зависимости, границы

**Главный принцип:** режь по **вертикали** (одно законченное поведение пользователя), а не по слоям (вся база сразу).

## /goal — держать агента на цели (v2.1.139+)

```
/goal Все тесты в test/auth проходят (npm test выходит с кодом 0),
файл package.json не изменён — или остановись после 20 ходов
```

- Агент работает ход за ходом сам, без «продолжай»
- После каждого хода маленькая модель (Haiku) проверяет условие
- **Условие нужно формулировать так, чтобы доказательство попадало в переписку**
- Ограничитель «или остановись после N ходов» — обязателен

## RALPH loop — для долгих задач

**Суть:** bash-цикл, каждый круг — новая сессия с чистым контекстом. Состояние на диске.

```bash
while :; do cat PROMPT.md | claude -p --dangerously-skip-permissions; done
```

- `PROMPT.md` — инструкция, не меняется между кругами
- Состояние: код, список задач (fix_plan.md), история Git
- `--dangerously-skip-permissions` → только в изолированном окружении

## Когда что использовать

| Инструмент | Контекст | Стоп |
|------------|----------|------|
| `/goal` | Короткие задачи, одна сессия | Проверяльщик подтвердил условие |
| `RALPH loop` | Долгие задачи (часы-дни), свежий контекст каждый круг | План пуст |
| `Stop hook` | Нужна своя логика проверки | Твой скрипт |

## Discovery Interview — скилл /discovery-interview

5 фаз:
1. Ориентация (2–3 вопроса)
2. Разбор по категориям (проблема, сценарии, данные, технологии, безопасность)
3. Research-петли (не уверен → Claude исследует, возвращается)
4. Разрешение конфликтов («быстро И на вырост» — выбери приоритет)
5. Проверка полноты по чек-листу

**Правила:** минимум 10–15 вопросов. После 3–5 ответов спеку не писать — получится «слоп».

## Антипаттерны (как НЕ отвечать на Discovery)

- «Просто простое X» → пока не определено что значит «простое» — агент угадает
- Технология без понимания цены → сначала зачем, потом чем
- «Как принято / как у всех» → перекладывает решение на агент = угадывание
- Описание решения вместо проблемы → из проблемы решений может быть несколько
- Спека написана и забыта → если ТЗ не обновлять, оно начинает врать агенту

---

## Конфликты требований — их нужно ловить явно

| Конфликт | Почему не сочетается |
|----------|---------------------|
| Простое И функциональное | Каждая фича усложняет интерфейс |
| Реальное время И дёшево | Мгновенные обновления = дорогая инфра |
| Безопасное И без трения | Каждая проверка — лишний шаг |
| Быстро собрать И на вырост | Гибкая архитектура требует времени сейчас |
